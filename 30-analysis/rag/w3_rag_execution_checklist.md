# OwlClaw RAG 数据层实施清单（12 周）

基线决策：`LlamaIndex + Qdrant + 二阶段重排（FlashRank/rerankers）`  
参考 ADR：`w3_rag_data_adr.md`

## 1) 角色分工模板

| 角色 | 核心职责 | 周交付物 |
|---|---|---|
| Tech Lead | 架构把关、技术风险清单、里程碑评审 | 周架构决策记录、风险闭环 |
| Data Engineer | 数据接入、清洗、切分、元数据规范 | 数据流水线状态、失败样本清单 |
| Retrieval Engineer | 索引策略、召回策略、混合检索、重排 | 离线评测报告、参数版本 |
| Backend Engineer | 检索 API、缓存、配额、租户隔离 | API 变更说明、SLA 报告 |
| MLOps/SRE | 部署、监控、告警、容量与成本治理 | 监控面板、容量报告、事故复盘 |
| QA/Analyst | Query 集建设、回归、A/B、上线门禁 | 质量门禁报告、回归结论 |
| PM | 需求优先级、验收标准、跨团队协调 | 周目标对齐、里程碑确认 |

## 2) 全局 Definition of Done

- 检索质量：离线 NDCG@10、Recall@k、MRR 达到阈值
- 性能：线上 P95 延迟满足 SLA（召回+重排全链路）
- 稳定性：索引任务成功率、回滚成功率达标
- 可运维：关键指标可观测（QPS、命中率、索引耗时、成本）
- 合规：依赖许可证完成审查并有记录

## 3) 12 周执行计划（按周）

### Week 1: 范围冻结 + 数据资产盘点
- [ ] 确认首批业务域与问答场景（FAQ/知识库/工单等）
- [ ] 盘点数据源与权限（文档库、DB、对象存储）
- [ ] 定义元数据协议：`doc_id/chunk_id/source/version/tenant_id/lang`
- [ ] 输出质量门槛初稿（Recall/NDCG/延迟/SLO）

验收：
- [ ] 有“首批场景清单”和“数据源清单”
- [ ] 元数据协议评审通过

### Week 2: 数据接入与清洗流水线 v1
- [ ] 接入最关键 2-3 个数据源
- [ ] 建立去重、清洗、分段（chunk）基础规则
- [ ] 记录失败样本并建立重试策略
- [ ] 建立 ingest 日志规范

验收：
- [ ] 每日增量任务可稳定跑通
- [ ] 失败率与原因可观测

### Week 3: 索引基线（Qdrant）+ 检索 API v1
- [ ] 建立 collection 规划（按租户/域/语言）
- [ ] 建立基础向量召回 API
- [ ] 设置索引参数基线（HNSW、top_k、过滤策略）
- [ ] 建立索引版本号与回滚机制

验收：
- [ ] API 可返回可解释结果（含来源元数据）
- [ ] 索引切换/回滚脚本可执行

### Week 4: 离线评测集与基线分数
- [ ] 建立 Query-Answer-GroundTruth 样本集（>=200）
- [ ] 打通离线评测流水线（Recall/MRR/NDCG）
- [ ] 建立“版本-指标”看板
- [ ] 冻结 v1 基线

验收：
- [ ] 指标可复现
- [ ] 基线版本可回放

### Week 5: 二阶段重排接入
- [ ] 接入 `FlashRank` 或 `rerankers`
- [ ] 设定召回候选池大小（如 top50/top100）
- [ ] 对比“仅召回 vs 召回+重排”
- [ ] 形成成本-效果曲线

验收：
- [ ] NDCG@10 提升达到目标（建议 >=10%）
- [ ] 延迟增量可控并可配置开关

### Week 6: 混合检索（可选 OpenSearch/Typesense）
- [ ] 接入关键词召回通道（BM25）
- [ ] 合并排序策略（向量分+词法分+重排分）
- [ ] 为长尾 query 做回退策略
- [ ] 完成灰度开关

验收：
- [ ] 长尾 query 命中率提升
- [ ] 混合策略可按租户配置

### Week 7: 多租户与安全隔离
- [ ] 租户级索引/权限/配额策略
- [ ] 请求级审计字段（tenant/request/model/version）
- [ ] 敏感字段脱敏与日志采样
- [ ] 合规检查单

验收：
- [ ] 越权测试通过
- [ ] 审计日志可追溯

### Week 8: 线上灰度与A/B
- [ ] 选择 1-2 条业务流量灰度
- [ ] A/B 对比（命中质量、延迟、成本）
- [ ] 观察失败模式并修复
- [ ] 决策是否扩大流量

验收：
- [ ] A/B 报告完成并结论明确
- [ ] 回滚预案演练通过

### Week 9: 可观测性与告警完善
- [ ] 指标：QPS、P95、召回命中率、重排耗时、索引时延
- [ ] 告警：失败率突增、数据新鲜度超时、延迟超阈
- [ ] 仪表盘与周报自动化
- [ ] 关键链路 Trace

验收：
- [ ] 重大告警可 5 分钟内定位
- [ ] 周报自动产出

### Week 10: 成本治理
- [ ] 设定查询预算和限流策略
- [ ] 缓存策略（query/result/embedding）
- [ ] 热冷分层存储（高频/低频）
- [ ] 计算单千次查询成本

验收：
- [ ] 单位成本下降趋势可证明
- [ ] 不影响核心 SLA

### Week 11: 稳定性压测与灾备
- [ ] 并发压测、索引重建压测
- [ ] 故障注入（依赖超时、节点故障）
- [ ] 备份/恢复演练
- [ ] 容量扩展预案

验收：
- [ ] 压测报告通过阈值
- [ ] 恢复时间满足目标

### Week 12: 上线评审与版本发布
- [ ] 完成发布检查单（质量/性能/安全/合规）
- [ ] 形成运维手册与值班手册
- [ ] 完成知识转移（KT）
- [ ] 发布 v1 正式版

验收：
- [ ] 发布评审通过
- [ ] 生产观测稳定一周

## 4) 周会模板（建议）

- 本周目标完成率（%）
- 指标变化：Recall/NDCG/P95/成本
- 风险 Top3 与负责人
- 下周里程碑与依赖项

## 5) 升级触发条件（从中规模到大规模）

满足任一条件，启动 `Milvus + OpenSearch` 迁移评估：
- 单集群数据量 > 100M chunks
- 峰值 QPS 持续超阈且扩容边际变差
- 多地域高可用成为硬需求
- 混合检索复杂度超出当前栈维护能力
