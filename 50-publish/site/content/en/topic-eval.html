Evaluation and Observability

Why it matters

Without eval gates and trace coverage, teams ship prompt and tool changes blind. The result is silent quality drift and slow incident recovery.

Key links in this repo

  • Eval and observability analysis
  • Release gate rules (YAML)
  • Rollback runbook
  • KPI catalog (CSV)

Key findings

  • Trace coverage and dataset versioning are the true baseline, not model choice.
  • Eval signals must be tied to promotion rules and rollback triggers.
  • SQL-based eval is useful for drift detection when logs are normalized.

Evidence sources

  • openai/evals
  • langfuse/langfuse
  • truera/trulens

Risks

  • Metric sprawl without ownership leads to false confidence.
  • Sampling bias if eval sets are not refreshed and rebalanced.
  • Observability gaps across tool boundaries hide regressions.

Next actions

  • Define promotion gates: data + eval + reliability + cost thresholds.
  • Require trace contracts for new tools and gateways.
  • Schedule drift reviews tied to release windows.
