RAG and Data Stack

Why it matters

RAG quality depends on retrieval architecture, evaluation loops, and operational fit.

Key findings

  • Vector store choice should follow workload shape, not popularity.
  • Evaluation design often limits quality more than model choice.
  • Metadata strategy strongly impacts retrieval reliability.

Evidence sources

  • run-llama/llama_index
  • qdrant/qdrant
  • chroma-core/chroma

Risks

  • Overfitting retrieval parameters to narrow benchmark sets.
  • High migration cost when index schema is tightly coupled.
  • Silent quality regressions without stable eval sets.

Next actions

  • Adopt a fixed evaluation set and periodic drift checks.
  • Separate retrieval logic from storage-specific adapters.
  • Track quality-cost-latency tradeoffs in one dashboard.

RAG Data and Retrieval

Why it matters

In many production assistants, user trust is primarily determined by retrieval quality, not by model size. Weak retrieval makes every downstream layer look unstable.

Key findings

  • Default stack should be profile-based, not one-size-fits-all.
  • Two-stage retrieval is high leverage but should be conditional by query complexity.
  • Upgrade planning must include migration rehearsal and compatibility checks.

Evidence sources

  • run-llama/llama_index
  • qdrant/qdrant
  • milvus-io/milvus
  • opensearch-project/OpenSearch

Risks

  • Static quality thresholds can be gamed by query mix shifts.
  • Reranker always-on can add cost with marginal gain.
  • Index rollback may fail if embedding/version compatibility is ignored.

Next actions

  • Adopt stratified eval gates by intent and difficulty.
  • Enable uncertainty-driven reranking instead of blanket reranking.
  • Run dual-read migration drills before major storage upgrades.
