Security and Governance

Why it matters

Agent systems amplify risk through tool invocation, context injection, and weak policy boundaries.

Key findings

  • Input/output checks are necessary but not sufficient.
  • Tool permission boundaries and audit traces are the real control layer.
  • Security controls must be tested under realistic multi-step workflows.

Evidence sources

  • guardrails-ai/guardrails
  • protectai/rebuff
  • openai/openai-agents-python

Risks

  • False sense of safety from output-only guardrails.
  • Privilege escalation when tool scopes are not explicit.
  • Missing evidence trails for incident response and governance.

Next actions

  • Define mandatory tool scope policy and enforcement checks.
  • Attach security controls to release gates, not post-release audits.
  • Track security regression scenarios in weekly challenge cycles.

Security and Governance

Why it matters

Agent platforms expand attack surface across prompt ingress, tool invocation, and generated artifacts. Security must be enforced as runtime control, not documentation.

Key findings

  • Policy engine, auth graph, and sandbox should be designed as one control stack.
  • Binary pass/fail KPIs are insufficient without adversarial coverage metrics.
  • Supply-chain trust needs SBOM, signature, and provenance gates together.

Evidence sources

  • open-policy-agent/opa
  • openfga/openfga
  • e2b-dev/E2B
  • sigstore/cosign

Risks

  • Zero-incident metrics can hide detection blind spots.
  • Archived/no-license corpora can introduce legal uncertainty.
  • Exception paths can silently bypass policy without expiry controls.

Next actions

  • Add breakout-test cadence and adversarial coverage index.
  • Enforce time-bound exception policy with signed approvals.
  • Publish legal handling rules for benchmark and corpus ingestion.
