Security and Governance

Why it matters

AI systems introduce new attack surfaces: prompt injection, tool abuse, data leakage, and privilege escalation through integrations. Governance must be implementable.

Key links in this repo

  • Governance analysis
  • Control-module mapping
  • Mapping table (CSV)
  • Risk register

Key findings

  • Tool boundary security is a default requirement, not an enhancement.
  • Redaction, timeouts, and path validation prevent common data leakage failures.
  • Exception policies must be timeboxed and enforceable by automation.

Evidence sources

  • Azure/PyRIT
  • openai/evals
  • guardrails-ai/guardrails

Risks

  • Policy documents drift if they are not tied to runtime gates.
  • Over-permissive tool access turns prompt injection into action injection.
  • Observability gaps hide sensitive data leaks.

Next actions

  • Enforce allowlists, redaction, and timeouts at tool boundaries.
  • Adopt a threat model per integration lane (MCP, gateway, RAG, eval).
  • Require exception expiry automation before production promotion.
