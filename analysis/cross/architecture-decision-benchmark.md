# Architecture Decision Benchmark（ADR 风格）

## ADR-000：基线说明
- **Status**: Proposed
- **Date**: 2026-03-09
- **Scope**: `agent / mcp / rag / gateway / eval`
- **Decision Drivers**:
  - 可落地速度（2-6 周内可上线）
  - 多供应商可移植性（避免单点锁定）
  - 可观测性与可审计性（生产治理）
  - 成本/性能平衡（QPS、延迟、Token 成本）
  - 组织复杂度（团队是否具备运维能力）

---

## ADR-001：Agent 编排赛道（5 选）
- **Decision**: 在生产里优先“可控状态机 + 可观测 + 可回滚”的框架。
- **Why**: Agent 系统失败通常不是“不会做”，而是“无法稳定重复做”。

| Project | Trade-off（取舍） | 适用边界 | 失败条件 | 推荐场景 |
|---|---|---|---|---|
| LangGraph | 强状态流与持久化，开发复杂度高于纯 prompt 编排 | 需要多步骤、有状态、可中断恢复流程 | 状态爆炸、节点定义混乱导致维护失控 | 金融/客服/审批等可追溯流程 Agent |
| CrewAI | 多角色协作上手快，但复杂控制与调试成本上升 | 中等复杂任务拆分，团队偏应用开发 | 角色职责重叠、循环 delegating 导致效率崩塌 | 内容生产、研究助理、运营自动化 |
| AutoGen | 对话式多 Agent 灵活，生产治理需额外补齐 | 研发探索、复杂协作策略验证 | 对话回合不可控引发成本飙升与延迟失控 | 原型期多 Agent 实验与策略搜索 |
| OpenAI Agents SDK | 与模型生态结合紧密，跨厂商可移植性相对弱 | 以 OpenAI 生态为主、追求快速上线 | 供应商能力变化导致行为漂移 | 小团队快速构建工具调用 Agent |
| Semantic Kernel | 企业集成能力强，工程化偏重、学习曲线更陡 | .NET/企业 IT 体系、合规优先 | 插件/内存策略不统一导致链路脆弱 | 企业内流程自动化、Copilot 类应用 |

**Agent 赛道建议**
- 默认：`LangGraph`（生产主编排）+ 其他框架用于实验。
- 若团队偏业务快迭代：先 `CrewAI`，再收敛到状态化框架。

## ADR-002：MCP/工具协议赛道（5 选）
- **Decision**: 以开放协议做工具层解耦，避免 Agent 与工具实现强耦合。
- **Why**: 工具接入规模化后，协议与权限模型比“单工具能力”更关键。

| Project | Trade-off（取舍） | 适用边界 | 失败条件 | 推荐场景 |
|---|---|---|---|---|
| modelcontextprotocol/servers | 生态广、复用快；质量参差需治理 | 快速引入通用工具能力 | 直接引入未审计 server 导致安全风险 | 快速搭建企业工具目录 |
| fastmcp | 开发者体验好，但需自行建设生产级治理 | 自建 MCP server、快速迭代 | 缺少认证/审计/限流导致平台风险 | 内部工具服务标准化输出 |
| mcp-gateway（社区实现） | 统一入口好处大，新增网关层运维复杂 | MCP Server 数量多、需统一策略 | 网关单点故障或策略错误全局放大 | 多团队共享 MCP 工具平台 |
| smithery（MCP registry 生态） | 发现成本低，外部依赖与可信度需评估 | 需要“市场式”工具发现 | 供应链风险与版本漂移不可控 | PoC/探索期快速试错 |
| apisix + mcp adapter（组合） | 企业级网关能力强，集成成本较高 | 需强认证、流控、审计的 MCP 平台 | 适配层维护不足，协议升级时失配 | 生产级 MCP 控制面建设 |

**MCP 赛道建议**
- 默认：`MCP Servers + 自建网关治理层`。
- 强合规组织：优先“API Gateway 化 MCP”（认证、审计、配额先行）。

## ADR-003：RAG 赛道（5 选）
- **Decision**: 采用“向量库 + 检索编排 + 评测闭环”的三段式。
- **Why**: RAG 失败多数源于召回与评测缺失，不是模型本身。

| Project | Trade-off（取舍） | 适用边界 | 失败条件 | 推荐场景 |
|---|---|---|---|---|
| Qdrant | 向量检索性能与过滤能力平衡，需运维索引策略 | 中大规模语义检索 + metadata filter | 索引参数不当导致召回/延迟劣化 | 企业知识库、检索问答 |
| Weaviate | 功能丰富（schema/模块），运维复杂度较高 | 需要混合搜索与丰富 schema | schema 演进失控导致查询不稳定 | 多租户知识平台 |
| Milvus | 大规模性能强，集群治理门槛高 | 亿级向量与高吞吐 | 资源与参数调优不足导致成本过高 | 大规模检索基础设施 |
| Elasticsearch（Vector） | 关键词与向量一体化，向量极限性能非最优 | 已有 ES 体系、需混合检索 | 查询 DSL 复杂导致维护困难 | 日志/文档检索融合场景 |
| LanceDB | 本地化与轻量化友好，分布式能力边界明确 | 单机/小规模、边缘或开发环境 | 数据规模增长后迁移成本骤增 | 本地 Agent、桌面知识检索 |

**RAG 赛道建议**
- 默认：`Qdrant` 作为生产基线；已有 ES 资产重时选 `Elasticsearch`。
- 强调：必须配套 offline/online eval，不做“盲 RAG 上线”。

## ADR-004：Inference Gateway 赛道（5 选）
- **Decision**: 采用“LLM 专用网关 + 服务网格弹性 + 成本账本”的组合。
- **Why**: 仅靠单网关很难同时解决路由、SLA、成本与审计。

| Project | Trade-off（取舍） | 适用边界 | 失败条件 | 推荐场景 |
|---|---|---|---|---|
| LiteLLM | LLM 路由/预算能力强，企业治理需补外部系统 | 多模型多供应商统一入口 | 只做网关不做账本导致成本失真 | AI API 统一代理与策略路由 |
| Portkey | 产品化体验好，部分高级能力依赖平台形态 | 希望快速落地治理能力 | 过度依赖单平台导致迁移难 | 中小团队快速治理上线 |
| Kong AI Gateway | 企业网关能力成熟，LLM 特性需方案化组合 | 强认证/配额/审计需求 | 配置复杂导致变更风险高 | 大企业 API/AI 一体治理 |
| Apache APISIX | 高性能 + 开放扩展，需更多自研整合 | 追求高吞吐与自定义插件 | 插件策略失配引发线上行为偏差 | 自主可控 API/AI 网关 |
| Istio（mesh） | 韧性治理强，学习与运维成本最高 | 大规模微服务与统一流量治理 | 团队无 mesh 经验导致复杂度反噬 | 核心生产集群 SLA 护栏 |

**Gateway 赛道建议**
- 默认：`LiteLLM + Istio + OpenMeter`（功能、韧性、账本三层）。
- 若团队小：`Portkey` 先跑通，再逐步替换为可控组件。

## ADR-005：Eval/Observability 赛道（5 选）
- **Decision**: 采用“Tracing + LLM 评测 + 业务指标”三线并行。
- **Why**: 只有可观测没有评测，无法判断“回答是否真的更好”。

| Project | Trade-off（取舍） | 适用边界 | 失败条件 | 推荐场景 |
|---|---|---|---|---|
| Langfuse | LLM 原生 tracing/eval 体验好，需治理采样与隐私 | 需要 prompt/trace/eval 一体化 | 采样策略不当导致成本过高或盲区 | 生产 Agent 可观测与迭代 |
| Helicone | API 观测与成本分析强，深度评测能力需组合 | 先解决成本与请求画像 | 只看成本不看质量导致错误优化 | 网关层成本与延迟治理 |
| Arize Phoenix | 评测与分析能力强，接入链路需工程投入 | 重视模型质量评估 | 评测集老化导致结论失真 | RAG/Agent 质量回归体系 |
| OpenLIT | OTel 友好、开放度高，产品化程度相对低 | 想统一到 OTel 体系 | 无统一语义约定导致数据碎片化 | 统一可观测平台集成 |
| Promptfoo | 离线评测简单直接，在线闭环需自建 | CI/CD 门禁与回归测试 | 用例集不足导致“测而无效” | 发布前回归、模型对比 |

**Eval 赛道建议**
- 默认：`Langfuse + Promptfoo`（线上追踪 + 线下门禁）。
- 若组织已有可观测平台：引入 `OpenLIT/OTel` 做统一汇聚。

## 横向决策结论（跨赛道）

## 推荐组合 A（生产优先，稳态治理）
- Agent: `LangGraph`
- MCP: `MCP Servers + 自建网关层`
- RAG: `Qdrant`
- Gateway: `LiteLLM + Istio + OpenMeter`
- Eval: `Langfuse + Promptfoo`

**适用边界**
- 中大型团队、对 SLA/成本/审计有硬要求。

**主要 trade-off**
- 前期工程投入较大，但中后期可控性最强。

**失败条件**
- 没有明确 owner 与 SLO，导致“组件齐全但无人治理”。

---

## 推荐组合 B（快速上线，低组织负担）
- Agent: `CrewAI` 或 `OpenAI Agents SDK`
- MCP: `托管/社区注册 + 最小网关`
- RAG: `Elasticsearch(Vector)`（复用存量）
- Gateway: `Portkey`（先跑通）
- Eval: `Helicone + Promptfoo`

**适用边界**
- 小团队、强交付时效、可接受阶段性平台依赖。

**主要 trade-off**
- 上线快，但后续可能面临迁移与深治理重构。

**失败条件**
- 用户量增长后仍维持 PoC 架构，出现成本和稳定性失控。

---

## 采纳与落地门槛（ADR 执行条件）
- 任何赛道选型必须同时满足：
  - 有清晰 owner（平台/ML/SRE/业务）
  - 有可量化 SLO 与失败回滚条件
  - 有替代方案与迁移预案（避免单点锁定）
- 若不满足以上 3 条，不进入生产。
