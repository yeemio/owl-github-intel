# AI开发平台研究问题树（500题）

说明：共500题，分5批，每批100题。每题包含“简答 + 下一步调查方向”。

## 第1批（Q001-Q100）：战略、产品定位与用户价值

- Q001 问题：AI开发平台的核心用户是谁？简答：通常是平台工程师、AI应用工程师与业务技术负责人三类。下一步：按角色拆分痛点访谈并建立RACI责任图。
- Q002 问题：平台首年最该解决什么问题？简答：优先解决“从原型到上线”断层，而非追求全功能。下一步：量化当前PoC转生产的失败率与平均周期。
- Q003 问题：平台如何定义“开发体验好”？简答：标准是低学习成本、低配置复杂度、可调试可回放。下一步：设计DX评分卡并做季度基准测试。
- Q004 问题：平台定位应该偏“工具箱”还是“工作台”？简答：早期应以工作台为主、工具箱可插拔。下一步：对比两种定位下的集成成本与迁移成本。
- Q005 问题：平台最小可行范围(MVP)怎么界定？简答：至少包含模型接入、编排、观测、发布四件套。下一步：把四件套拆成上线门槛清单。
- Q006 问题：为什么多数AI平台在二期失速？简答：因价值指标只看模型效果，忽略交付效率与治理能力。下一步：重建KPI体系，加入工程与治理指标。
- Q007 问题：平台应先服务内部还是外部客户？简答：建议先内部打磨，再外部扩展。下一步：建立内部“设计伙伴”机制和SLA试点。
- Q008 问题：平台如何避免“重复造轮子”？简答：用协议边界和开源适配层吸收生态能力。下一步：绘制自研/复用决策矩阵并设评审门禁。
- Q009 问题：平台品牌叙事该强调什么？简答：强调可控、可审计、可扩展，而非“最强模型”。下一步：统一对内外叙事口径并验证市场反馈。
- Q010 问题：平台价值如何向管理层解释？简答：用交付周期缩短、故障率下降、单位成本下降三指标。下一步：建立季度经营看板并复盘ROI。
- Q011 问题：是否要支持低代码人群？简答：可支持，但不应牺牲工程可治理性。下一步：定义低代码与专业开发的边界与协同流程。
- Q012 问题：平台上线节奏按什么划分？简答：按能力层分批：接入层、编排层、治理层、商业层。下一步：形成四阶段里程碑与验收标准。
- Q013 问题：平台成功的第一信号是什么？简答：业务团队愿意把关键流程迁入并持续使用。下一步：追踪周活项目数与留存率。
- Q014 问题：失败的早期信号有哪些？简答：功能多但无人深用、工单持续升高、迁移反复。下一步：建立红旗告警阈值并周度审查。
- Q015 问题：平台怎样平衡速度与稳定性？简答：开发环境追快、生产环境追稳，通过分级策略实现。下一步：实施环境分级与发布节流规则。
- Q016 问题：平台是否需要行业模板？简答：需要，模板能降低试错和沟通成本。下一步：优先做3个高频行业模板并测复用率。
- Q017 问题：平台对业务方最直观的价值是什么？简答：需求到上线时间可预测、质量可追踪。下一步：发布“需求到上线”周期月报。
- Q018 问题：如何定义平台“可规模化”？简答：并发、租户、模型、团队四维都能线性扩展。下一步：做四维压测并形成容量曲线。
- Q019 问题：平台何时该开放API优先？简答：当内部流程稳定后，应以API作为主交付面。下一步：梳理核心API清单与版本策略。
- Q020 问题：UI优先还是SDK优先？简答：研发场景优先SDK，运营场景补UI。下一步：统计调用路径占比指导投入。
- Q021 问题：平台能否成为组织标准件？简答：可以，但前提是协议统一与职责清晰。下一步：推动架构委员会确认平台标准地位。
- Q022 问题：平台初期最常见误区是什么？简答：把“模型能力”当成“平台能力”。下一步：列出平台能力分层并对齐组织认知。
- Q023 问题：何时需要平台产品经理角色？简答：当跨团队依赖超过三条主链路时必须配置。下一步：建立平台需求分级和优先级机制。
- Q024 问题：平台如何支撑多业务线？简答：采用“共享底座+业务扩展包”模式。下一步：设计扩展包规范和审核流程。
- Q025 问题：平台如何做需求治理？简答：按“复用潜力+风险等级+收益”三维排序。下一步：上线需求评分看板并每两周评审。
- Q026 问题：平台如何处理长尾需求？简答：提供扩展点，不把长尾硬编码进核心。下一步：定义插件协议和兼容性测试。
- Q027 问题：平台需要专门的开发者关系吗？简答：需要，内部开发者同样需要文档与赋能。下一步：建立开发者教育路线图。
- Q028 问题：平台度量应偏结果还是过程？简答：两者都要，过程指标预警，结果指标验收。下一步：构建双层指标体系。
- Q029 问题：平台该如何设定北极星指标？简答：可用“生产有效发布次数/季度”。下一步：定义“有效发布”的统一口径。
- Q030 问题：平台怎么减少认知负担？简答：统一术语、统一配置形态、统一错误语义。下一步：建立平台词汇表和错误码规范。
- Q031 问题：平台要不要内置知识库能力？简答：应内置基础能力，复杂场景走外接。下一步：划分内置与外接能力边界。
- Q032 问题：如何判断平台是否“过度设计”？简答：若新增能力三个月无真实业务使用即偏过度。下一步：建立能力下线机制。
- Q033 问题：平台目标客户是否只限技术团队？简答：不是，还包括运营、风控、合规与客服。下一步：补齐非研发角色旅程图。
- Q034 问题：平台化与项目制如何协同？简答：项目制验证价值，平台化沉淀共性。下一步：设立“项目到平台”迁移流程。
- Q035 问题：平台是否该提供预置评测集？简答：是，能显著降低上线争议。下一步：建设基础评测集并持续版本化。
- Q036 问题：平台如何证明“不是成本中心”？简答：把复用率和节省工时转化为财务语言。下一步：建立平台收益核算模型。
- Q037 问题：平台是否需要灰度机制？简答：必须需要，AI行为波动决定了灰度是刚需。下一步：定义灰度分层和回滚策略。
- Q038 问题：如何确保平台路线不过度跟风？简答：以业务问题驱动，而非模型热点驱动。下一步：建立季度路线评审模板。
- Q039 问题：平台中台化会不会拖慢业务？简答：会，若审批链过长且标准不清。下一步：优化审批SLA并引入自动化检查。
- Q040 问题：平台何时适合做商业化？简答：当内部复用稳定、能力边界清晰且可复制时。下一步：试点一个外部客户包。
- Q041 问题：平台如何处理“试验性功能”？简答：采用实验区隔离，不直接进入核心主干。下一步：建立实验功能生命周期。
- Q042 问题：平台如何支持跨区域团队？简答：通过多区域部署与统一控制面。下一步：验证跨区延迟与数据合规策略。
- Q043 问题：平台为何需要统一身份系统？简答：身份是权限、审计、计费的共同基座。下一步：梳理身份主数据来源与映射。
- Q044 问题：平台何时该引入成本预算控制？简答：从第一天就应有预算阈值。下一步：按团队/应用建立预算与告警。
- Q045 问题：平台是否要做模型中立？简答：应尽量中立，降低供应商锁定风险。下一步：用适配器抽象提供商差异。
- Q046 问题：平台能力如何做优先级排序？简答：先“阻断项”，后“增益项”。下一步：列出当前Top10阻断项并逐一拆解。
- Q047 问题：平台如何支持敏捷迭代？简答：用小步快跑+标准回滚+可观测闭环。下一步：把每次发布绑定回滚演练。
- Q048 问题：平台为何要有“弃用策略”？简答：没有弃用策略就会被历史包袱拖垮。下一步：发布API弃用时间表。
- Q049 问题：平台是否应提供推荐配置？简答：应提供，避免用户从零配置。下一步：沉淀按场景的默认配置集。
- Q050 问题：平台如何衡量用户满意度？简答：看NPS还不够，要看任务完成率。下一步：建立“任务成功调查”机制。
- Q051 问题：平台如何处理跨团队冲突需求？简答：用标准化评审与透明优先级。下一步：引入公开需求看板和裁决规则。
- Q052 问题：平台的“核心资产”是什么？简答：不是代码本身，而是可复用的能力模型。下一步：梳理能力资产目录。
- Q053 问题：平台如何缩短新团队上手时间？简答：模板化脚手架和逐步文档最有效。下一步：测量新团队首个上线时长。
- Q054 问题：平台需要多语言SDK吗？简答：视团队技术栈而定，但至少双语言。下一步：统计语言分布并规划SDK路线。
- Q055 问题：平台文档优先级应如何排？简答：先“上手+故障排查”，后“原理细节”。下一步：重构文档目录与入口。
- Q056 问题：平台如何定义“可运维性”？简答：可定位、可回放、可回滚、可追责。下一步：建立运维能力验收清单。
- Q057 问题：平台怎样避免“过度抽象”？简答：抽象必须由3个以上真实场景驱动。下一步：抽象前提交场景证据。
- Q058 问题：平台是否应支持离线开发？简答：应支持部分离线，提高研发连续性。下一步：定义离线模拟能力范围。
- Q059 问题：平台如何管理变更风险？简答：按风险分级发布与审批。下一步：建立变更风险评分模型。
- Q060 问题：平台能否统一实验记录？简答：可以，通过实验追踪与版本关联实现。下一步：把实验记录接入发布流程。
- Q061 问题：平台如何处理“工具过载”？简答：减少入口、统一编排、按场景打包。下一步：清理重复工具并合并入口。
- Q062 问题：平台如何管理“隐性依赖”？简答：依赖图可视化并强制登记关键依赖。下一步：上线依赖扫描与告警。
- Q063 问题：平台怎么做能力分层定价？简答：按算力、调用量、治理能力分层。下一步：建立内部计费模拟账单。
- Q064 问题：平台是否应支持插件市场？简答：中后期可做，前期先做内部插件库。下一步：制定插件安全与质量标准。
- Q065 问题：平台如何避免“单点专家”风险？简答：流程化、文档化、自动化替代口口相传。下一步：识别单点环节并去风险。
- Q066 问题：平台需要业务成功团队吗？简答：对内也需要，确保能力真正落地。下一步：建立应用落地复盘机制。
- Q067 问题：平台如何形成组织共识？简答：以事实数据驱动，而非概念争论。下一步：发布月度平台影响报告。
- Q068 问题：平台如何处理跨代模型切换？简答：通过兼容层和回归集保障平滑切换。下一步：建立模型升级演练流程。
- Q069 问题：平台是否必须支持多租户？简答：企业场景通常必须支持。下一步：先从逻辑隔离做起，再到物理隔离。
- Q070 问题：平台怎样支持A/B实验？简答：编排层和网关层都应可切流。下一步：定义实验策略模板与统计方法。
- Q071 问题：平台价值如何映射到业务KPI？简答：映射到转化、成本、时效和风险四类指标。下一步：建立业务KPI映射表。
- Q072 问题：平台如何防止“影子系统”出现？简答：提供足够灵活扩展点和快速响应机制。下一步：盘点影子系统并回迁。
- Q073 问题：平台的迭代节奏应多快？简答：核心能力双周，关键稳定能力月度。下一步：按能力类型设发布节拍。
- Q074 问题：平台是否应做“最佳实践中心”？简答：应做，能减少重复犯错。下一步：建设案例库和反模式库。
- Q075 问题：平台如何提升跨团队协作效率？简答：标准接口与统一工单流程是关键。下一步：建立跨团队SLO。
- Q076 问题：平台该如何做路线透明？简答：公开路线图和变更日志。下一步：定期发布Roadmap评审纪要。
- Q077 问题：平台是否应支持混合云？简答：对大型企业通常是必须项。下一步：评估混合云下数据路径和成本。
- Q078 问题：平台在组织中归谁负责？简答：平台技术负责人牵头，业务联合治理。下一步：明确治理委员会职责。
- Q079 问题：平台如何降低试点失败成本？简答：模板化试点与可撤销架构设计。下一步：定义试点退出条件。
- Q080 问题：平台要不要提供“专家模式”？简答：要，用于高级调参与调试。下一步：设计专家模式的权限与审计。
- Q081 问题：平台怎样管理需求波峰？简答：需求分级和容量预留并行。下一步：建立需求高峰应急机制。
- Q082 问题：平台如何做版本兼容承诺？简答：明确LTS与快线双轨。下一步：发布版本支持策略文档。
- Q083 问题：平台为何要做错误分级？简答：错误分级决定响应优先级和处理路径。下一步：统一错误分级标准。
- Q084 问题：平台如何提高可发现性？简答：能力目录、示例仓库、可搜索文档三位一体。下一步：上线能力导航中心。
- Q085 问题：平台如何处理“功能请求泛滥”？简答：以问题为中心，不以功能为中心。下一步：改造需求模板为问题模板。
- Q086 问题：平台应否内置治理策略模板？简答：应内置，可降低合规落地难度。下一步：先做权限、审计、数据保留模板。
- Q087 问题：平台如何支撑国际化团队？简答：多语言文档、时区友好运维、区域化部署。下一步：制定全球化支持计划。
- Q088 问题：平台该如何推进组织采纳？简答：先拿下标杆业务再复制。下一步：选择两条高影响业务线试点。
- Q089 问题：平台如何识别“伪需求”？简答：看是否可转化为可验证业务结果。下一步：需求立项前做结果假设评审。
- Q090 问题：平台能否降低人才门槛？简答：通过标准化与自动化可显著降低。下一步：测算不同资历人群上手时间。
- Q091 问题：平台如何建立信任？简答：稳定性、透明度、可回滚能力共同构成信任。下一步：公开发布稳定性报告。
- Q092 问题：平台为什么需要“反模式清单”？简答：反模式比最佳实践更能避免代价。下一步：沉淀Top20反模式案例。
- Q093 问题：平台如何处理遗留系统接入？简答：通过适配层渐进接入，避免一次性重构。下一步：制定遗留接入优先级。
- Q094 问题：平台是否要设“创新沙盒”？简答：要，创新应隔离在可控环境。下一步：定义沙盒资源和退出规则。
- Q095 问题：平台能否支持端到端责任追踪？简答：可以，前提是统一链路ID与审计模型。下一步：落实全链路trace-id规范。
- Q096 问题：平台如何减少跨工具切换成本？简答：统一入口和上下文传递机制。下一步：统计切换次数并优化主流程。
- Q097 问题：平台为什么要重视变更沟通？简答：变更沟通不足会放大上线风险。下一步：建立变更公告与订阅机制。
- Q098 问题：平台如何定义“高质量需求”？简答：需求必须有目标、范围、约束、验收。下一步：上线需求质量评分。
- Q099 问题：平台如何规划未来两年能力图谱？简答：按“稳定核心+扩展生态”双线推进。下一步：绘制两年能力雷达图。
- Q100 问题：平台首批100天成功标准是什么？简答：能稳定支撑3个生产场景并形成可复制模板。下一步：制定100天行动与复盘计划。

## 第2批（Q101-Q200）：平台架构、编排与工程效率
